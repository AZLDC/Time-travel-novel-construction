# 專案導言(作者的話)

- 通過發想與AI對話，建構包含：大綱、設定和故事內容
- 透過因果變更紀錄來記錄事件的發生、結果，並且在後續內容中引用參考
- 透過劇情收束資訊和設定集，產生AI人物畫集，遴選之後，產生動畫和劇情互動等(放置，後續處置)
- 作品開始須明確描述該故事皆為虛構，乃是架空時空，劇中所有人物皆不存在，如有雷同，那你肯定穿越了，這本可是高武和輕玄幻的作品，就連時間單位都不一樣，不可能雷同，當然若你們誰穿書到我這來了，記得給個日記，俺想觀摩觀摩。


# AI 索引（L1 快速入口 + L2 詳細索引）

此檔案同時包含：
- L1：快速入口
- L2：詳細索引（完整 ID 表格與事件素材池）

### 使用原則（資料收束與 Context 管理）

- 本專案為長篇小說，**每章目標字數約 10,000+，總章數目標約 3,000+**。若直接依賴 `Story/` 內文作為記憶，將很快超出 AI 的 context 能力。
- 為此，作者已事先將長期需要記住的世界觀、人物關係、因果變更、章節摘要等，**預先「壓縮／濃縮／收束」到 `Settings/` 與 `Cards/` 檔案中**。
- AI 在本專案中工作時，應遵守以下原則：
  - 初次進入或懷疑記憶不足時：依 `README.md` 指引，先閱讀本檔案「快速入口」與對應章節卡 `Cards/cardNNNN.md`，再依本檔「讀取指南」決定是否展開其他 Settings 檔案。
  - **撰寫新章正文時**：以本檔案「快速入口」+ 本檔案索引區 + 對應章節卡 `Cards/cardNNNN.md` 為主要資訊來源，可視需要少量查閱 `因果紀錄.md`、`事件備案.md` 等；原則上不通讀 `Story/` 既有章節。
  - **修改既有正文時**：僅在明確需要修改舊文時，才打開對應 `Story/chNNNN_章名.md`，並盡量只閱讀與本次修改相關的段落以及其前後少量內容，以控制 context 使用量。
  - 本檔與 `Cards/` 中的內容，已是經過收束處理的「工作記憶」。**使用時不應再無差別刪減或過度摘要整份檔案**，而是直接依檔內規則與索引行動，必要時再回頭查 L3 詳細設定。

## 專案目錄結構

```
格物治世/
├── Story/                 ← 章節正文（chNNNN_章名.md，例如 ch0001_賜婚.md）
├── Settings/              ← 設定檔案
│   ├── AI索引.md          ← 本檔案（L1 快速入口 + L2 索引層）
│   ├── 人物設定.md
│   ├── 架空世界設定.md
│   ├── 故事設定或大綱.md
│   ├── 因果紀錄.md
│   └── 事件備案.md
├── Cards/                ← 各章規劃卡（cardNNNN.md，例如 card0001_賜婚.md）
└── _testPic/              ← 圖片素材（測試中未來功能，尚未規畫完成）
```

### 章節檔案命名規則

- **正文檔（Story）**：`Story/chNNNN_章名.md`
  - 例如：`ch0000_序章.md`、`ch0001_賜婚.md`、`ch0002_新婚.md`
- **章節卡（規劃卡）**：`Cards/cardNNNN.md`
  - 例如：`card0000.md`、`card0001.md`
- `NNNN` 為四位數章碼：序章=0000，第一章=0001，第二章=0002，依此類推。
 - 若章名中包含符號（如空格、頓號、破折號、括號等），**僅在 Story 檔名中的「章名」部分以底線 `_` 取代符號**，以避免檔名與路徑問題；章節卡檔名維持 `cardNNNN.md` 格式，實際章名請在檔案內容中使用完整寫法

**重要**：
- 章節正文一律存放於 `Story/` 目錄
- 設定檔案一律存放於 `Settings/` 目錄
- 章節卡（規劃用）存放於 `Cards/`（根目錄）

寫第 N 章時，依序完成（此處 N 對應四位數章碼 NNNN，如第1章→0001、第10章→0010）：

**【動筆前】**
1. [ ] 讀取前兩章的「章節卡」摘要（例如：本章編號為 0008 時，前兩章的章節卡為 `Cards/card0006.md`、`Cards/card0007.md`；若本章為 0001 則僅有 `Cards/card0000.md`）

4. [ ] 確認活躍人物可用性（查 `因果紀錄.md`「角色當前所在地」）

**【定稿後】**

## 章節完成後檢查機制

> ⚠️ **定稿後必須逐項檢查，確保設定一致性。**

**【世界觀與設定一致性】**
1. [ ] **發明/技術前置**：檢查本章出現的發明是否符合「發明索引」的前置條件與狀態
   - 例：銀鏡需玻璃前置，玻璃需高溫爐前置
   - 若發明尚未「生效」，不得在正文中讓非主角角色憑空使用或提及
2. [ ] **市場/商品存在性**：檢查正文提及的商品是否符合世界設定
   - 例：若玻璃鏡是主角首創，市面上不應存在「舶來水鏡」等同類商品
   - 比價對象應為當時已存在的物品（如銅鏡）
3. [ ] **人物可用性**：檢查本章出場人物是否在「角色當前所在地」標記為可用
   - 不可用角色（如邊關的父兄）不得在京城場景出現
4. [ ] **現代用語規範**：檢查是否違反「語言與現代知識使用規則」
   - 現代網路用語僅限主角內心 OS，或已觸發「現代用語外溢事件」後的少數丫鬟
   - 現代科技知識需先在因果紀錄標記「已向 X 角色解釋 Y 概念」

**【伏筆與因果一致性】**
5. [ ] **伏筆觸發合理性**：本章觸發的伏筆是否符合「伏筆索引」的預計觸發時機
6. [ ] **因果鏈連貫性**：本章事件是否與前章因果紀錄相容，無矛盾

## 風格索引（給 AI）

- **規範**：本文章預計為一般向，可能擦邊但不會過線，如果不行最多撤到輔導級，但應該不會。
- **章節格式**：
  - 章標題使用 `# 第N章　章名`
  - 段落間使用 `---` 分隔，**不使用小節標題**（如 `## 一、...`）
  - **不要結尾標記**（如 `（第N章完）`）
- **整體文體**：現代中文網文敘事，偏輕鬆商戰／日常流，使用第三人稱近距視角，適度貼近主角或女主的心理。
- **基調**：
  - 以「甜寵日常」為主線氛圍，穿插「格物創新、商業操作、家族/朝堂鬥嘴」。
  - 衝突多為言語與立場衝突（文鬥、家族壓力、階級觀念），避免血腥暴力描寫。
  - 以劇情和對話帶出敘事，儘量減少平直的敘事感。
- **節奏**：
  - 一個場景內兼顧：生活感細節（吃飯、院子、衣物）＋情緒互動（撒嬌、護短）＋至少一個推進主線的小事件。
  - 章內允許多個小起伏，但每一小節應有清楚的「起→合」收束點。
- **語氣與對話**：
  - 對話口吻貼近日常口語，不過度文言。
  - 主角陳子鑑：吐槽、打趣、偶爾一本正經裝逼，有現代人內心 OS，但說話時會顧及時代語感。
  - 藍盈盈：前期怯懦、輕聲細語，逐漸變得有主見，但面對藍家仍有本能畏縮。
  - 隨因果紀錄嚴格控制現代用語與現代知識的使用：
    - 對話中**禁止出現明顯現代網路用語**（如「撒狗糧、CP、flag、BUG」等），男主可在內心 OS 使用現代詞彙，但不得讓旁人直接聽見；其他角色一律不得說出口。
    - 預計於第9章安排「現代用語外溢事件」：男主一時口快說出現代詞，下人學去在背後嘻笑使用，盈盈問起含義，男主敷衍解釋為「外地俚語（例如：當眾秀親熱給旁人看）」；**自該事件之後**，內宅少數丫鬟可在輕鬆場景偶爾以既定少量詞彙吐槽主角夫妻，但仍應節制使用，不可氾濫。
    - 現代科技／科學知識（如細菌、蒸汽機、電、火車等）原則上僅允許出現在男主的內心思考或格物實驗規劃中；若要向其他角色明說，必須先在 `因果紀錄.md` 以伏筆或事件形式明確標記「已向某角色解釋某概念」，未標記前，**禁止任何非主角角色憑空說出超越時代的技術名詞與關鍵原理**。
- **甜寵表現方式**：
  - 所有「主角×女主」同場景預設近距離接觸，能靠在一起就不特意拉開距離。
  - 對話自動偏甜寵語氣，適度加入害羞、心跳、耳紅等反應，但不寫露骨情慾獨白。
  - 動作優先設計為牽手、擁抱、摟抱、托起等親密肢體互動，仍避免超齡或露骨情慾描寫。
  - 任意事件皆可優先轉化為糖點橋段，默認甜度標準為「不黏一起不算夫妻」。
  - 若有衝突情節，一律以秀恩愛、互相選擇與信任收束為最高優先，避免撕裂感情線。
  - 女性旁觀者看到主角夫妻公開撒糖時，可有嘴上吐槽或不好意思，但情緒基調以羨慕、被塞狗糧為主；如有不適，多半選擇轉移視線或離開現場，而非主動攻擊。
  - 男性旁觀者多半以尷尬、吐槽「哇別這樣」或乾脆躲開來回應；少數對此表示欽佩。即便不喜歡，也傾向遠離而非公開發作。
